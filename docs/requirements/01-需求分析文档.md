## 1. 执行摘要

openGecko 是面向**开源社区运营团队**的企业级多租户内容管理与运营平台。系统以"社区"为第一公民，让社区项目经理（PM）进入系统后即可感知当前社区的整体运营状态，并从同一工作台完成内容创作、多渠道发布、活动管理、人脉运营、委员会治理、会议管理等日常工作。

**核心价值主张**：Manage All, Publish Everywhere。

**当前已交付能力**（Phase 1-4 全量交付）：
- **多租户架构**：共享数据库 + community_id 行级隔离，三级 RBAC（superuser / admin / user）
- **内容全生命周期管理**：上传 / 编辑 / 审核工作流 / 排期日历 / 多人协作 / 素材库 / 设计任务
- **四渠道发布**：微信公众号（API 自动）、Hugo（文件生成）、CSDN / 知乎（格式复制）
- **活动管理**：活动 CRUD、SOP 模板、任务分工、签到名单、反馈与 Issue 关联
- **人脉与 Campaign**：人脉档案、社区角色、Campaign 触达活动
- **生态洞察**：生态项目管理、数据快照、贡献者分析、洞察仪表板
- **社区治理**：委员会管理、会议管理（ICS 导出）、个人工作台
- **数据与安全**：微信统计、发布记录追踪、审计日志、渠道凭证 Fernet 加密、消息通知

## 2. 业务背景与问题陈述

### 2.1 业务背景

开源基金会通常同时运营多个开源社区（如 openEuler、openGauss、MindSpore 等），每个社区都有独立的内容产出需求、发布渠道账号、活动运营和治理结构。现有运营痛点包括：

| 痛点 | 具体表现 |
|------|---------|
| 多渠道发布效率低 | 同一篇文章需手动复制粘贴到微信、CSDN、知乎，格式调整耗时 |
| 内容状态不可见 | 多人协作时无法统一追踪哪些内容在审核、哪些已排期 |
| 活动管理散乱 | 活动 SOP、任务分工、签到名单、反馈追踪分散在各工具 |
| 人脉数据孤岛 | 贡献者信息分散在委员会、活动、生态项目等不同系统中 |
| 社区数据分散 | 跨社区的内容产出、发布效果数据无法统一分析 |
| 治理协作低效 | 委员会成员信息和会议安排散落在各处文档和电子表格 |
| 权限边界不清 | 不同社区管理员不应互相看到对方数据，但缺少系统级隔离 |

### 2.2 解决方案定位

openGecko 以"社区"为核心组织单元，提供：
1. **统一登录入口**：所有社区数据在同一平台管理，切换社区自动切换数据视角
2. **以 PM 工作流为中心的设计**：内容策划 → 协作编辑 → 审核发布 → 活动管理 → 治理管理，全流程贯通
3. **人脉中台**：`PersonProfile` 作为统一人脉数据库，所有模块共享同一人员数据
4. **数据安全隔离**：`community_id` 强过滤，普通用户绝不可见其他社区数据

## 3. 用户角色与画像

### 3.1 角色层级

```
Platform Level
└── Superuser（平台管理员）：跨社区全权限，创建/删除社区，工作量总览

Community Level
├── Admin（社区管理员）：社区内全权限，管理成员，配置渠道
└── User（普通成员）：编辑自己负责的内容，参与活动
```

### 3.2 典型用户画像

| 角色 | 核心诉求 | 常用功能 |
|------|---------|---------|
| **社区运营 PM** | 统观全局，指挥内容和活动 | 社区总览、内容日历、工作量总览、Campaign |
| **内容创作者** | 高效创作，一键多发 | 内容编辑器、素材库、渠道发布 |
| **活动负责人** | 标准化流程，全程追踪 | 活动管理、SOP 模板、签到、反馈 |
| **委员会成员** | 掌握会议与任务动态 | 个人工作台、会议管理、ICS 导出 |
| **开发者关系** | 了解贡献者、策划触达 | 人脉档案、生态洞察、Campaign |
| **平台管理员** | 系统健康、安全合规 | 用户管理、审计日志、渠道配置 |

## 4. 已实现功能模块详述

### 4.1 身份认证与用户管理

**需求描述**：安全的用户认证体系，支持初始化引导流程，避免默认密码长期暴露。

**已实现功能**：
- JWT Bearer Token 认证，Token 过期可刷新
- 首次启动引导：默认临时账号（`DEFAULT_ADMIN_USERNAME`）登录后强制创建正式管理员，创建完成后临时账号自动删除
- 密码重置：SMTP 邮件发送一次性 Token 链接，Token 24 小时有效
- 用户信息管理：个人资料编辑，修改密码

**页面**：`Login.vue`、`ForgotPassword.vue`、`ResetPassword.vue`、`InitialSetup.vue`、`Profile.vue`

### 4.2 多租户社区管理

**需求描述**：支持多个独立社区在同一平台共存，数据严格隔离。

**已实现功能**：
- 社区 CRUD（名称、Slug、描述、Logo）
- 成员邀请与角色分配（admin / user）
- 社区切换（`X-Community-Id` Header）
- 社区向导（初次创建流程引导）
- 社区沙箱（测试环境）

**页面**：`CommunityManage.vue`、`CommunityOverview.vue`、`CommunitySettings.vue`、`CommunityWizard.vue`、`CommunitySandbox.vue`

### 4.3 内容管理

**需求描述**：支持从创作到发布全流程，多人协作，可视化排期。

**已实现功能**：
- 内容 CRUD：标题、摘要、正文（Markdown）、封面图
- 文件上传：`.docx` / `.md` 上传后自动转换为 Markdown，提取图片
- 在线编辑器：md-editor-v3，支持实时预览、插入素材
- 内容状态工作流：`draft → reviewing → approved → published`
- 工作状态：`planning / in_progress / completed`（独立字段）
- 所有权模型：`created_by_user_id`（不可变）+ `owner_id`（可转让）+ collaborators + assignees
- 内容归属多社区：`content_communities` 关联表（支持跨社区内容）
- 日历视图：FullCalendar 按 `scheduled_publish_at` 可视化排期，支持拖拽
- 封面图管理：上传并设置内容封面，发布微信时自动作为封面

**页面**：`ContentList.vue`、`ContentEdit.vue`、`ContentCalendar.vue`

### 4.4 素材库

**需求描述**：统一管理可复用的图片和文档素材，支持编辑器内插入。

**已实现功能**：
- 素材上传（图片、文档）
- 按类型过滤（`image` / `document`）
- 编辑器内一键插入素材链接
- 社区隔离（每个社区独立的素材库）

**页面**：`AssetLibrary.vue`

### 4.5 设计任务

**需求描述**：内容配图需求管理，追踪设计师工作进度。

**已实现功能**：
- 设计任务 CRUD（关联内容、任务类型、优先级、截止日期）
- 分配设计师（`assignee_id`）
- 任务状态追踪（`pending / in_progress / completed / cancelled`）
- 按社区、状态、优先级、类型过滤

**页面**：`DesignTasks.vue`

### 4.6 多渠道发布

**需求描述**：内容一次创建，多渠道分发，记录每次发布结果。

**已实现功能**：

| 渠道 | 实现方式 | 特殊处理 |
|------|---------|---------|
| 微信公众号 | 调用微信 API 创建草稿 | 自动上传图片至微信 CDN，替换图片 URL |
| Hugo 博客 | 生成带 front matter 的 `.md` 文件 | 写入指定本地博客仓库路径 |
| CSDN | 返回格式化内容供复制 | 适配 CSDN Markdown 格式 |
| 知乎 | 返回格式化内容供复制 | 适配知乎编辑器格式 |

- 渠道配置：AppID/Secret/Cookie 等凭证 Fernet 加密存储
- 发布记录：记录每次发布的渠道、时间、状态、错误信息
- 发布统计：按渠道维度汇总

**页面**：`PublishView.vue`

### 4.7 活动管理

**需求描述**：支持线上/线下/混合活动的全生命周期管理，可复用 SOP 模板。

**已实现功能**：
- 活动 CRUD：名称、类型、时间、地点、描述、多社区关联
- SOP 模板库：
  - 模板 CRUD（名称、描述、关联社区）
  - 模板项（Checklist Item）：支持 `is_mandatory`、`responsible_role`、`deadline_offset_days`、`estimated_hours`、`reference_url`
  - 从模板创建活动时自动生成 Checklist
- 活动 Checklist：独立追踪每项任务状态（`pending / in_progress / done`），`completed_at` 自动记录
- 活动人员分工：主持人（`host`）、嘉宾（`speaker`）、志愿者（`volunteer`），可关联人脉档案
- 活动签到名单：`EventAttendee` 支持关联人脉档案
- 活动反馈：收集反馈意见，关联 GitHub Issue（`IssueLink`）
- 活动日历：按时间轴展示

**页面**：`Events.vue`、`EventDetail.vue`、`EventTemplates.vue`

### 4.8 人脉与 Campaign

**需求描述**：建立统一的贡献者/联系人数据库，支持精准触达策划。

**已实现功能**：

**人脉档案（PersonProfile）**：
- 统一人员信息：姓名、邮箱、公司、职位、GitHub/Gitee handle
- 技术标签、社区归属
- 关联委员会成员、活动人员、生态贡献者

**社区角色（CommunityRole）**：
- 记录一个人在不同社区中的角色（贡献者、Maintainer、TSC 成员等）
- 支持多个任期（`start_date` / `end_date`）

**Campaign（触达活动）**：
- Campaign CRUD：名称、目标、状态（`planning / active / completed / cancelled`）
- 支持多个 owner（`owner_ids` JSON 字段）
- Campaign 联系人（`CampaignContact`）：关联人脉档案，独立状态（`pending / contacted / responded / converted / closed`）
- 接触活动记录（`CampaignActivity`）：记录每次接触的渠道、内容、时间
- 联系人状态简化：`pending → contacted → responded → closed`

**页面**：`People.vue`、`PeopleDetail.vue`、`Campaigns.vue`、`CampaignDetail.vue`

### 4.9 生态洞察

**需求描述**：追踪社区生态中的开源项目健康度，分析贡献者群体特征。

**已实现功能**：
- **生态项目**：CRUD（名称、仓库 URL、语言、License、描述、健康分）
- **数据快照**：定期采集 Star/Fork/Issue/PR/贡献者数，保存历史快照（`EcosystemSnapshot`）
- **自动同步配置**：支持配置同步间隔（`auto_sync_enabled`、`sync_interval_hours`）
- **贡献者管理**：`EcosystemContributor` 记录每个项目的贡献者，可关联人脉档案
- **Insights 仪表板**：
  - 企业影响力分析（贡献者所属企业分布）
  - 贡献者趋势图
  - 项目健康度对比

**页面**：`EcosystemList.vue`、`EcosystemDetail.vue`、`InsightsDashboard.vue`

### 4.10 微信数据统计

**需求描述**：对微信公众号发布的文章进行阅读量等数据的录入与分析。

**已实现功能**：
- 文章统计录入（`WechatArticleStat`）：阅读数、点赞数、转发数
- 按文章分类（`article_category`）聚合统计
- 趋势图：按天/周/月汇总（`WechatStatsAggregate`）
- 社区维度隔离

**页面**：`WechatStats.vue`、`Analytics.vue`

### 4.11 社区治理

**需求描述**：管理委员会组织结构和会议，提升治理透明度。

**已实现功能**：

**委员会（Committee）**：
- CRUD：名称、Slug、类型、描述
- 成员管理（`CommitteeMember`）：姓名、邮箱、角色、任期，可关联人脉档案
- 多届委员会支持

**会议（Meeting）**：
- CRUD：标题、委员会、议程、纪要、附件、地点类型（online / offline / hybrid）
- 状态追踪：`scheduled → in_progress → completed → cancelled`
- 工作状态：`planning / in_progress / completed`
- 参与者管理（`MeetingParticipant`）
- 分配人（`MeetingAssignee`）
- ICS 日历导出
- 邮件提醒（`MeetingReminder`，`reminder_type = before_24h / before_1h`）

**页面**：`CommitteeList.vue`、`CommitteDetail.vue`、`CommitteeMemberManage.vue`、`MeetingDetail.vue`、`MeetingCalendar.vue`、`GovernanceOverview.vue`

### 4.12 个人工作台 & 数据仪表板

**需求描述**：用户快速了解自己的工作状态；管理员掌握全局情况。

**已实现功能**：
- **个人工作台**（`/my-work`）：跨社区汇总我负责的内容（assignee）、我参与的会议，按工作状态分组
- **社区总览**（Community Dashboard）：内容产出统计、成员数、近期会议、发布趋势、微信统计
- **工作量总览**（`/workload`）：superuser 专属，跨社区对比各成员内容数量和工作状态
- **个人仪表板**（`/dashboard`）：审核中内容数、代办事项、近期活动提醒

**页面**：`Dashboard.vue`、`MyWork.vue`、`WorkloadOverview.vue`

### 4.13 消息通知

**需求描述**：系统内通知，让用户及时了解与自己相关的事件。

**已实现功能**：
- 通知 CRUD（`Notification`）：标题、内容、类型、关联资源
- 已读/未读状态
- 未读数量统计（导航栏徽标）
- 批量标记已读

### 4.14 平台管理

**需求描述**：superuser 专属的跨社区管理工具。

**已实现功能**：
- **用户管理**：查看所有用户，重置密码，激活/禁用账号，分配超级管理员
- **审计日志**：记录所有关键操作（操作类型、资源、用户、社区、时间）
- **渠道配置**：各社区的微信/Hugo/CSDN/知乎渠道凭证（Fernet 加密存储）
- **平台设置**：SMTP 配置、功能模块开关（`ENABLE_INSIGHTS_MODULE`）

**页面**：`UserManage.vue`、`AuditLogs.vue`、`Settings.vue`

## 5. 非功能需求

### 5.1 性能
- API 响应时间：P95 < 500ms（非文件上传场景）
- 并发：支持 100 并发用户（单实例 Gunicorn 4 worker）
- 文件上传：最大 10MB，超时 60s

### 5.2 安全
- 认证：JWT Bearer Token，访问令牌 30 分钟过期
- 密码：bcrypt 哈希，最低 8 位
- 渠道凭证：Fernet 对称加密，密钥由 `JWT_SECRET_KEY` 派生，API 响应脱敏
- SQL 注入防护：SQLAlchemy ORM 全量参数化，禁止拼接 SQL
- XSS 防护：Vue 3 模板自动转义，`v-html` 仅用于受信内容
- CSRF 防护：JWT 在 Authorization Header，无 Cookie 认证
- 文件上传：MIME 类型白名单，随机文件名防路径穿越

### 5.3 可靠性
- 数据库迁移：Alembic 版本管理，所有迁移均可回滚
- 错误处理：全局异常捕获，统一 `{"detail": "..."}` 响应格式
- 审计日志：关键操作全量记录，不可删除

### 5.4 可维护性
- 测试覆盖率：后端 ≥ 80%（CI 强制检查）
- 代码质量：ruff lint + black 格式化 + mypy 类型检查
- 前端类型安全：Vue 3 + TypeScript strict 模式

### 5.5 可扩展性
- 多租户：新增社区无需改代码，仅数据配置
- 渠道扩展：新增发布渠道只需在 `services/` 添加一个 service class
- 功能开关：`ENABLE_INSIGHTS_MODULE` 等环境变量控制模块启停

## 8. 验收标准

### 8.1 功能验收

| 功能域 | 验收标准 |
|--------|---------|
| 认证 | 登录/登出/密码重置流程可正常完成；临时账号在首次创建正式管理员后自动删除 |
| 多租户隔离 | A 社区用户调用任何 API 均不可获取 B 社区数据 |
| 内容工作流 | draft→reviewing→approved→published 状态流转按权限规则执行 |
| 微信发布 | 包含本地图片的 Markdown 内容发布后，图片 URL 已替换为微信 CDN 地址 |
| 活动 SOP | 从模板创建活动后，自动生成对应 Checklist；勾选完成时 `completed_at` 正确记录 |
| 数据安全 | 渠道凭证 API 响应中不出现明文 Secret/Cookie |
| 通知 | 相关操作后目标用户收到系统通知，未读数正确显示 |

### 8.2 非功能验收

## 9. 术语表

| 术语 | 含义 |
|------|------|
| Community | 一个独立的开源社区，是系统最高级的数据隔离单元 |
| Superuser | 平台管理员，拥有跨社区全权限 |
| Admin | 社区管理员，社区内全权限 |
| User | 普通成员，仅能操作自己负责的内容 |
| Content | 一篇内容（文章），有完整的工作流生命周期 |
| Owner | 内容的当前负责人，可转让 |
| Assignee | 内容的执行负责人（可多人），用于工作台汇总 |
| Collaborator | 内容协作者，可编辑但非 owner |
| Channel | 发布渠道（微信/Hugo/CSDN/知乎），凭证加密存储 |
| PersonProfile | 统一的人脉档案，贡献者/联系人的标准数据源 |
| Campaign | 一次有目标的人脉触达活动 |
| SOP Template | 活动的标准操作程序模板，包含可复用 Checklist |
| Fernet | 对称加密算法，用于渠道凭证加密 |
| community_id | 所有社区范围数据的隔离键 |
| X-Community-Id | 前端请求 Header，指示当前操作所属社区 |
